# 02. 핵심 개념 (Core Concepts)

이 문서는 MDA의 **개념적 구조**를 정의한다. 데이터 모델(`03_DATA_MODEL.md`)의 전제가 되는 문서.

---

## 2.1 재귀 만다라트 (Recursive Mandarart)

### 기본 구조

만다라트는 **3x3 격자 × 2단계 = 9x9 격자**가 전통적 형태다. 중심 목표 → 8개 영역 → 각 영역 8개 실행.

MDA는 이를 **3계층 데이터 모델**로 매핑한다:

```
Mandarart (Root) = 중심 목표
├── Section 1 (8개 중 하나)
│   ├── Task 1-1 (8개 중 하나)
│   ├── Task 1-2
│   ├── ... (최대 8개)
│   └── Task 1-8
├── Section 2
│   └── Tasks...
...
└── Section 8
```

### 재귀 확장

실제 프로젝트는 3계층으로 부족할 수 있다. **Task가 다시 Mandarart이 될 수 있다** (재귀).

```
Project "제주 미션북"
├── Section "기획" 
│   ├── Task "컨셉 확정" ← 작은 태스크
│   ├── Task "타겟 분석" (복잡함) → 하위 Mandarart로 전환
│   │   ├── Section "타겟 정의"
│   │   ├── Section "시장 조사"
│   │   └── ...
│   └── ...
├── Section "제작"
└── ...
```

### "8개" 제약의 유연성

경직성 완화를 위해:
- **최소**: 1개 (섹션/태스크가 하나여도 무방)
- **권장**: 3~6개
- **최대**: 8개 (만다라트 철학)
- **초과 시**: AI가 자동으로 "재귀 만다라트 생성" 제안

### 12개월 같은 자연스러운 초과

분기 → 월 구조로 해결:
```
Annual Project
├── Q1 (Section)
│   ├── Jan (Task → 하위 Mandarart)
│   ├── Feb (Task → 하위 Mandarart)
│   └── Mar (Task → 하위 Mandarart)
├── Q2, Q3, Q4
```

### 완성 판정

- Task가 완료되면 Section의 진행률 증가
- Section 내 모든 Task 완료 시 Section 완성
- 모든 Section 완성 시 Project 완성
- 재귀 Task는 하위 Mandarart의 완성률이 자신의 상태

## 2.2 다축 피벗 (Multi-Axis Pivot)

### 기본 개념

같은 Project 데이터를 **다른 축으로 재배열**해서 보여주는 기능.

```
데이터 큐브:
  Task × (Section, Assignee, Time, Status, Deliverables)

축 전환 (버튼 하나):
- 섹션 축: Section별로 Task 그룹핑 → 기본 만다라트
- 팀 축: Assignee별로 Task 그룹핑 → "철수 만다라트", "영희 만다라트"
- 시간 축: 날짜/주/월별로 Task 그룹핑 → "1월 만다라트", "2월 만다라트"
```

### 피벗 가능 조건

모든 Task는 **완전한 메타데이터**를 가져야 한다:
- `assignee_id` (담당자)
- `due_date` or `time_block` (시간)
- `section_id` (섹션)
- `status` (상태)
- `deliverables[]` (산출물, optional)

**하나라도 빠지면** 해당 축에서 "미배정" 셀에 모임.

### 데이터 완전성 UX

메타데이터가 빠진 Task는:
- 만다라트에서 **회색 셀**로 표시 (시각적 부채)
- AI 에이전트가 리마인드: "담당자가 정해지지 않은 태스크가 3개 있어요"
- 사용자가 상호작용 시 **우선 보완 제안**

### 피벗 시각 언어

**만다라트 그리드는 유지**하되, 셀 라벨만 변경:
- 섹션 축: 중심셀 = 프로젝트명, 8셀 = 섹션명
- 팀 축: 중심셀 = 프로젝트명, 8셀 = 팀원명
- 시간 축: 중심셀 = 프로젝트명, 8셀 = 기간 (1월~8월, Q1~Q4 등)

## 2.3 멀티 뷰 전환 (Multi-View Switching)

### 4가지 뷰

같은 데이터(Task 큐브)를 4가지 시각 언어로 렌더링:

| 뷰 | 강점 | 기본 대상 |
|----|------|----------|
| **투두 리스트** | 빠른 체크, 플랫 | Light |
| **만다라트** | 전체-부분 동시 인지 | Medium |
| **캘린더** | 시간 배치 | 모든 페르소나 |
| **간트** | 의존관계·타임라인 | Heavy |

### 전환 메커니즘

```
User → [뷰 전환 버튼] → 같은 데이터, 다른 렌더러
                        ├── useSingleTaskSource() (데이터 훅)
                        ├── ListView / MandaraView / CalendarView / GanttView
                        └── 전환 시 상태 유지 (선택된 태스크, 필터 등)
```

### 페르소나별 기본 뷰

- Light: 투두 리스트 또는 **일일 만다라트** (오늘 태스크의 3x3)
- Medium: 프로젝트 만다라트 + 투두 투샷
- Heavy: 간트 또는 팀 만다라트

### 일일 만다라트 (Light 전용 특화)

하루 태스크 8개 이하면 그대로 3x3 격자. 8개 초과 시 시간 블록으로 분할:

```
아침     |  오전     |  점심
운동     |  회의     |  식사
         |          |
오후     |  저녁     |  밤
집중작업  |  장보기   |  일기 쓰기
         |          |
```

각 블록은 최대 3개 태스크. 즉 하루 최대 9개 깔끔하게 표현.

## 2.4 Dual Asset Branching (업무/일기 이중 분기)

### 입력 → 분기 → 출력

```
[INPUT]
  ├── 음성 녹음 (필요할 때 버튼)
  ├── 필기/타이핑
  ├── 파일 업로드
  ├── 통화 녹음 임포트 (OS에서 공유)
  └── 외부 연동 데이터 (Calendar/Gmail 등)
      ↓
[AI 의도 분류]
      ↓
      ├─→ [BUSINESS ASSET] — 프로젝트·태스크·캘린더
      │   ├── 독립 태스크 (project_id 없음)
      │   └── 프로젝트 태스크 (project_id 있음, 만다라트 반영)
      │
      └─→ [JOURNAL ASSET] — 감정·관계·일상
          ├── JournalEntry (일기 항목)
          ├── PersonContact (관계 업데이트)
          └── Reminder (만날 사람 리마인드)
      ↓
[EVENING SYNTHESIS]
      ↓
[DAILY REPORT] — 두 자산의 통합 뷰
```

### 독립 태스크 vs 프로젝트 태스크

**독립 태스크**:
- `project_id` 없음
- 만다라트에서는 "Inbox" 또는 미표시
- 캘린더·투두에만 등장
- Light 유저 기본 모드

**프로젝트 태스크**:
- `project_id` 있음
- 만다라트 셀에 반영
- 바텀업 집계에 기여
- Medium/Heavy 유저 기본 모드

**자연스러운 성장 경로**: 처음엔 독립 → AI가 "이 태스크들 묶으면 프로젝트네요" 제안 → 프로젝트로 전환

### AI 자동 매칭 확신도

사용자가 음성으로 "보고서 쓰기" 라고 하면 AI는:

1. 기존 활성 프로젝트·섹션 목록 조회
2. 매칭 확신도 계산 (의미론적 유사도)
3. 분기:
   - **90%+**: 자동 배치, 사용자에게 알림 ("Q1 마케팅 > 리포팅에 추가함")
   - **60-90%**: 제안만 표시, 사용자 승인 필요
   - **<60%**: 독립 태스크로 유지, 저녁 보고서에서 재검토

## 2.5 프로젝트 생명주기

### 생성 단계

1. **자연어 대화**: "1월에 마케팅 캠페인 해야 해"
2. **AI 러프 분해**: 3~5개 섹션 제시
3. **대화형 보완**: 담당자·일정·산출물 묻기 (1~2개씩)
4. **AI 확인받기**: 애매한 것만 사용자에게
5. **프로젝트 차트 생성**: 만다라트로 즉시 시각화

### 실행 단계

- 일일 리마인드: "오늘 3개의 태스크 예정"
- 진행 알림: "A 섹션 완료!"
- 리스크 감지: "B 태스크가 지연 중"
- 자동 재조정: "지연된 태스크를 다음 주로 미룰까요?"

### 완료 단계

- 섹션 완성 → 축하 애니메이션
- 프로젝트 완성 → 회고 제안
- 완료 프로젝트는 보관 (삭제 아님)

## 2.6 저녁 보고서 (Daily Synthesis)

### 구조

매일 정해진 시간(사용자 설정, 기본 21:00)에 AI가 자동 생성:

```markdown
# 2026-04-21의 하루

## 🏢 비즈니스 자산
### 오늘 완료 (3)
- ✅ 부산 인플루언서 컵 발송
- ✅ Q1 마케팅 리서치 마감
- ✅ 팀 스탠드업

### 오늘 진행
- 🔄 제주 미션북 프로젝트 킥오프

### 새로 추가됨 (2)
- ➕ 5월 1일 성당 (아버지 기일)
- ➕ 임 외 2명 카페 방문 대응

### 내일 예정 (4)
...

## 📓 일기 자산
### 오늘의 감정 곡선
아침: 평온 → 오후: 활기 → 저녁: 피곤

### 오늘의 한 줄
"가족과 보낸 오후가 좋았다. 미션북 시작의 첫 걸음."

### 만난 사람
- 임OO 외 2명 (카페 방문)

### 생각할 것
- 5월 1일 전후 일정 비워두기

## 🔔 리마인드
- 4/29, 4/30: 성당 준비
- 5/1: 아버지 기일
```

### 기능

- **원본 드릴다운**: 각 항목에서 원본 음성/텍스트로 복귀 가능
- **사용자 수정**: 잘못 분류된 것 1탭으로 이동
- **저장**: 그날의 보고서는 영구 보관 (회고 자산)

## 2.7 관계 관리 (Contact Intelligence)

### PersonContact 엔터티

음성·대화에서 **인물 언급**이 감지되면 자동으로 PersonContact 생성 또는 업데이트.

```
PersonContact {
  name: "임OO"
  relationship: "카페 단골"
  last_contact: "2026-04-21"
  context: ["4/21 카페 방문"]
  reminder: null | { date, reason }
}
```

### 리마인드 메커니즘

사용자가 "나중에 A 만나봐야겠다"라고 하면:
- AI가 "언제쯤 만날까요?" 질문
- 사용자 답변 시 `reminder` 설정
- 설정일 근처에 알림

## 2.8 맥락 수집 (Ambient Context)

### 의도적 녹음 vs 수동적 맥락

MDA는 **상시 감청을 하지 않는다**. 대신 두 경로로 맥락을 모은다:

**의도적 입력 (Active)**:
- 사용자가 녹음 버튼 누름
- 필기/타이핑
- 파일 업로드
- 통화 녹음 임포트 (OS 공유)

**수동적 맥락 (Passive, 사용자 동의 하)**:
- Google Calendar 이벤트
- Gmail 제목 (본문 아님, 선택적)
- 사진 라이브러리 메타데이터 (위치·시간)
- 위치 이력 (선택적)
- 건강 데이터 (걸음·수면, 선택적)

### 맥락 활용

- 저녁 보고서 생성 시 배경 재료
- "오늘 어디 다녀오셨네요" 같은 회고 질문
- 만난 사람 추정 (같은 시간대 캘린더 이벤트)

## 2.9 다꾸 시스템 (Decoration)

### 목적

**감정적 소유감**을 통한 리텐션. "내 공간"이라는 느낌.

### 커스터마이징 가능 요소

- **테마 프리셋**: 8~16개 기본 테마
- **컬러 팔레트**: 포인트 컬러 선택
- **스티커**: 기본 스티커팩 + 프리미엄 팩
- **셀 장식**: 테두리·배경·이모지
- **폰트 크기/스타일**: 3단계
- **완료 애니메이션**: 폭죽·스탬프·물결 등 선택

### 데이터 저장

- User 레벨: 전체 테마
- Project 레벨: 프로젝트 전용 테마 (override)
- Task 레벨: 셀별 커스텀 (emoji, color, sticker)
- Section/Cell 레벨: 섹션 배경 패턴

### Light vs Heavy 차등

- Light: 다꾸 UI 노출, 기본값이 꾸며진 상태
- Heavy: 다꾸 UI는 설정에만, 기본값이 조용한 상태
- 사용자 선택으로 어떤 페르소나든 상대 측 스타일 사용 가능

## 2.10 Frictionless Input

### 3버튼 위젯

홈화면·잠금화면·워치에 배치:

```
┌──────────────────────────┐
│   🎙️    ✏️    📎        │
│   녹음  필기  업로드     │
└──────────────────────────┘
```

### 각 버튼 동작

**🎙️ 녹음**: 탭 → 즉시 녹음 시작 → 탭 종료 → 자동 처리  
**✏️ 필기**: 탭 → 최소 입력 시트 → 입력 후 저장  
**📎 업로드**: 탭 → OS 파일 선택 → AI 처리

### 원칙

- 2탭 이내 입력 시작
- 앱 풀스크린 안 열기 (최소 시트)
- 입력 후 홈 자동 복귀
- AI 처리는 백그라운드
- 결과는 나중에 푸시 알림 또는 저녁 보고서

### 확장 (Phase 2+)

- Apple Watch 앱
- Wear OS 앱
- Siri/Google Assistant 단축어
- 공유 시트 통합 (다른 앱에서 MDA로 공유)

## 2.12 용어 정리 (Glossary)

| 용어 | 정의 |
|------|------|
| **Mandarart** | 프로젝트/섹션/태스크의 3x3 재귀 구조 |
| **Section** | 프로젝트의 하위 영역 (최대 8) |
| **Task** | 실행 단위. 독립 또는 프로젝트 소속 |
| **Pivot** | 같은 데이터의 축 변경 (섹션/팀/시간) |
| **View Mode** | 렌더링 방식 (투두/만다라트/캘린더/간트) |
| **Persona** | Light/Medium/Heavy — 사용자 프로필 |
| **Dual Asset** | 업무/일기 이중 자산 구조 |
| **Journal Entry** | 일기 항목 |
| **PersonContact** | 관계 관리 엔터티 |
| **Daily Report** | 저녁 자동 생성 보고서 |
| **Decoration (다꾸)** | 사용자 커스터마이징 요소 |
| **Intent Classification** | AI의 의도 분류 (업무/일기/질문 등) |
| **Active Input** | 사용자가 의도적으로 제공한 입력 |
| **Passive Context** | 연동 서비스에서 자동 수집된 맥락 |
