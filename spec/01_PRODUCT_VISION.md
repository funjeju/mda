# 01. 프로덕트 비전 & 철학

## 1.1 북극성

> **"나의 하루를 — 라이프든 비즈니스든 — 가장 스마트하고 편리하게, 치밀하고 불안하지 않게 요약하고 관리해주는 솔루션"**

### 한 문장 분해

| 구절 | 약속 |
|------|------|
| "나의 하루" | 프로덕트 범위 — 하루 단위가 기본 호흡. 프로젝트는 여러 하루의 누적. |
| "라이프든 비즈니스든" | 듀얼 자산 분기. 영역 구분 없이 다 받음. |
| "스마트하고 편리하게" | AI가 무거운 걸 대신. 사용자는 입력만. |
| "치밀하고" | 빠짐 없음. 업무 자산이 흘러나가지 않음. |
| "불안하지 않게" | 감정적 안전감. "놓쳤나?" 불안 제거. |
| "요약하고 관리" | 입력이 자산이 되고, 자산이 실행으로 이어짐. |

**핵심 역설 해결**: 기존 생산성 앱은 **치밀해질수록 불안해지거나** 반대로 **단순해지면 빠짐이 생겼다**. MDA는 AI가 치밀함을 짊어지고, 사용자에게는 정돈된 뷰만 제공해 이 역설을 해결한다.

## 1.2 기존 시장의 실패 지점

### 기존 투두/프로젝트 앱의 한계

| 카테고리 | 대표 앱 | 한계 |
|----------|---------|------|
| 라이트 투두 | Things, TickTick, Apple Reminders | 프로젝트 연결 약함, AI 없음 |
| 헤비 프로젝트 | Asana, Jira, Linear | 일상 잡무 부담, 라이트 유저에 과함 |
| 노트 기반 | Notion, ClickUp | 자유도 과잉으로 "뭘 해야 할지 모름" |
| AI 스케줄러 | Motion, Reclaim, Sunsama | 자동 배치 중심, 구조적 분해 없음 |
| 일기 | Day One, Diarly | 업무 연동 없음 |
| 기록 | Rewind, Limitless | 업무/감정 분류 없음 |
| AI 비서 | ChatGPT, Gemini | 대화만 남고 시각 자산 없음 |

### 공통 실패 원인

1. **사용자가 메타 작업(분류·태깅·스케줄링·리뷰)을 떠안음** → 앱은 단순한데 앱을 쓰기 위한 일이 복잡
2. **뷰가 경직됨** → 한 가지 시각 언어 강요 (리스트 or 칸반 or 캘린더)
3. **AI가 있다면 "입력 도우미" 수준** → 진짜 의사결정은 사용자 몫
4. **업무와 일상 분리** → 한 앱에서 둘 다 처리 불가
5. **기록과 실행 분리** → 일기는 일기대로, 할 일은 할 일대로

## 1.3 MDA의 해법 — 8대 설계 축

### 축 1. 구조적 분해 (만다라트 3단계 재귀)

```
Project (최종 목표)
  └── Section (8개 이하 하위 영역)
        └── Task (8개 이하 실행 항목)
              └── (재귀 가능: Task가 다시 Project가 될 수 있음)
```

- 만다라트를 **구조적 메타포**로 삼음 — 단순 UI가 아님
- 셀 완료 → 섹션 완성 → 프로젝트 완성 (바텀업 집계)
- 셀이 다시 하위 프로젝트가 되는 재귀 구조
- "8개 이하" 제약은 인간 인지 한계에 맞춤

### 축 2. 다축 피벗

같은 데이터를 **3개의 축**으로 전환 가능:

- **시간 축** — 1월/2월/… 또는 오전/오후/저녁
- **팀 축** — 담당자별 만다라트
- **섹션 축** — 프로젝트 논리 구조별

버튼 하나로 전환. 같은 시각 언어(만다라트) 유지로 **인지 부하 최소화**.

### 축 3. 듀얼 페르소나 + 멀티 뷰 전환

**페르소나**:
- **Light** — 일상 투두 중심, 다꾸 강조, 심플
- **Medium** — 개인 프로젝트 + 일상, 균형
- **Heavy** — 팀 프로젝트, 조용한 프로페셔널 톤

**뷰 모드** (사용자가 버튼 하나로 전환):
- 투두 리스트
- 만다라트
- 캘린더
- 간트 (Heavy 전용)

동일 데이터 모델 위에서 렌더러만 교체.

### 축 4. 앱+웹 듀얼 플랫폼

**앱 (iOS/Android)** — 일상 입력·알림·위젯의 1차 무대
- 홈/잠금 화면 위젯 (3버튼: 녹음/필기/업로드)
- 푸시 알림
- 음성 녹음 1급 시민

**웹 (Next.js)** — 헤비 설계·분석·협업의 1차 무대
- 프로젝트 설계 대화
- 대시보드 인사이트
- 팀 관리

### 축 5. Claude Code 주도 개발

- MD 문서가 개발의 진실 (Source of Truth)
- Claude Code가 MD 기반으로 구현
- 사용자(제작자)는 디렉터·아키텍트 역할

### 축 6. MCP/Skill로 약점 보강

- Figma MCP → 디자인 품질
- Firebase MCP → 스키마 검증
- GitHub MCP → 빌드·배포
- 자체 Skill → 도메인 지식·프롬프트 패턴

### 축 7. Frictionless 입력

**3버튼 위젯** (홈·잠금화면·워치):
- 🎙️ 녹음
- ✏️ 필기
- 📎 업로드

2탭 이내 입력. 앱 풀스크린 안 열고 최소 시트만.

### 축 8. Dual Asset Branching

**입력** (음성·텍스트·파일·통화 녹음 임포트)  
→ **AI 의도 분류**  
→ **업무 자산** (프로젝트·태스크·캘린더) / **일기 자산** (감정·관계·일상)  
→ **저녁 보고서** (두 자산 통합 대시보드)

## 1.4 페르소나

### Light User — "김민지, 27세, 마케터"

**특징**:
- 업무는 회사 툴(Jira)에서 하고, 개인적으로는 깔끔한 일상 관리가 필요
- 아침 스케줄·저녁 운동·친구 약속·병원 등 다양한 일상 태스크
- 꾸미는 걸 좋아함, SNS에 다꾸 스크린샷 공유
- 감정 일기도 쓰고 싶음

**사용 패턴**:
- 기본 뷰: 투두 또는 오늘의 만다라트 (3x3 시간 블록)
- 주로 독립 태스크 (project_id 없음)
- 음성 입력 활용도 높음
- 저녁 보고서 중요
- 다꾸로 감정적 소유감

**성공 지표**: 일일 접속, 다꾸 커스터마이징, SNS 공유

### Medium User — "박준호, 34세, 1인 프리랜서 디자이너"

**특징**:
- 클라이언트 프로젝트 3~5개 동시 진행
- 혼자 일하지만 프로젝트 단위 관리 필요
- 개인 일상도 병행 (운동·독서·가족)
- 디자인 툴·노션 사용 중

**사용 패턴**:
- 기본 뷰: 프로젝트 만다라트 + 오늘의 투두
- 프로젝트 태스크 + 독립 태스크 혼재
- 노션 임포트 활용
- 캘린더 연동 필수

**성공 지표**: 프로젝트 완성률, 지연 태스크 감소

### Heavy User — "이수현, 41세, 스타트업 PM"

**특징**:
- 팀 10명 관리, 복잡한 프로젝트 여러 개
- 일일 스탠드업·주간 리뷰·분기 OKR
- Jira·Slack·Notion 등 여러 툴 혼용 중 통합 필요
- 보고서·대시보드 중요

**사용 패턴**:
- 기본 뷰: 간트 또는 팀별 만다라트
- 모든 태스크가 프로젝트 소속
- 담당자 배정·알림·진행률 모니터링
- 대시보드 인사이트 활용

**성공 지표**: 팀 생산성, 프로젝트 예측 정확도, 팀원 온보딩 속도

## 1.5 경쟁 포지셔닝

### 2x2 매트릭스

```
               구조적 / 프로젝트 중심
                      │
                      │
    Asana, Linear ────┼──── MDA ★
                      │
                      │
   단순 ─────────────────────────────── 복잡
                      │
                      │
  Apple Reminders ────┼──── Notion, ClickUp
                      │
                      │
               자유로운 / 기록 중심
```

- **MDA = 구조적이면서 단순** — 모순처럼 보이지만 AI가 복잡도를 흡수함으로써 가능
- 경쟁자 중 이 사분면에 들어온 제품은 없음

### 차별화 요약

| 특성 | 기존 시장 | MDA |
|------|----------|-----|
| 구조화 | 사용자가 함 | AI가 함 |
| 뷰 | 고정 | 버튼으로 전환 |
| 자산 타입 | 업무 OR 일기 | 업무 AND 일기 |
| 입력 | 타이핑 중심 | 음성·필기·업로드 3버튼 |
| 결과물 | 리스트·텍스트 | 만다라트 + 보고서 |
| 치밀함 | 사용자 부담 | AI 부담 |

## 1.6 장기 비전

### v1 — Life OS for One (Phase 1~3)

개인이 하루를 치밀하고 편하게 관리하는 도구. 듀얼 자산 분기 + 만다라트 구조 + 팀 협업.

### v2 — Life OS for Teams

팀 단위 리추얼(일일 스탠드업·주간 회고) AI 자동화. 크로스 프로젝트 인사이트.

### v3 — Life OS Ecosystem

써드파티 스킬·자동화·API. 사용자가 자신만의 에이전트 행동을 정의.

### v4 — 도메인 특화 확장

- MDA Edu (교사용)
- MDA Creator (크리에이터용)

## 1.8 설계 원칙 (판단이 흔들릴 때)

1. **심플이 이긴다** — 기능을 더하는 것보다 빼는 것이 어렵다
2. **AI가 짊어진다** — 복잡도는 백엔드로
3. **뷰는 선택이다** — 강요하지 않는다
4. **원본은 성역이다** — AI 처리가 원본을 대체하지 않는다
5. **다꾸는 기능이다** — 감정 = 리텐션
6. **알림은 신호다** — 죄책감이 아니다
7. **치밀하되 불안하지 않게** — 모든 기능은 이 역설을 지키는지 검증
